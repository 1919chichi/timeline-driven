# TimeTrackr / PhaseDo 产品文档（按当前代码实现）

---

# 一、产品概述

## 1.1 产品定位

**TimeTrackr（PhaseDo）** 是一个聚焦「时间区间内每日执行」的轻量任务追踪工具。

当前实现不是传统意义上的待办清单，也不是无限循环的 Habit 工具，而是用于管理：

> 在某一段时间内，需要每天重复完成的任务

从当前界面与文案来看，产品明显偏向 **游戏代肝 / 多账号日常执行** 场景，但模型本身也可以用于学习计划、健身计划、短周期项目执行等场景。

---

## 1.2 当前版本核心价值

* 用开始日期和结束日期定义任务生效区间
* 将任务区分为「进行中」与「即将开始」
* 支持按分组组织任务
* 支持按“整任务”或“子标签计数”进行每日打卡
* 所有数据保存在本地，打开即用

---

## 1.3 当前信息架构

当前代码只有一个页面，没有路由系统：

* `今天（进行中）`
* `即将开始`
* `任务新建 / 编辑弹窗`

已结束任务当前不会出现在界面中。

---

# 二、核心数据模型

## 2.1 Task 模型

```json
{
  "id": 1712620800000,
  "name": "少年游",
  "group": "游戏代肝",
  "start": "2026-03-24",
  "end": "2026-04-24",
  "tags": [
    { "name": "协战", "max": 1 },
    { "name": "勾协", "max": 1 },
    { "name": "寄养", "max": 1 }
  ],
  "remark": "#359697 | 春区酪酪 | 邮箱",
  "logs": {
    "2026-04-09": {
      "协战": 1,
      "勾协": 1,
      "寄养": 0
    }
  }
}
```

---

## 2.2 字段说明

| 字段 | 含义 |
| --- | --- |
| `id` | 任务唯一标识，新建时使用时间戳生成 |
| `name` | 任务名称，当前 UI 文案为“游戏ID / 任务名称” |
| `group` | 所属分组，用于列表分区展示 |
| `start` | 开始日期，格式为 `YYYY-MM-DD` |
| `end` | 结束日期，可为空；为空时任务不会因超期自动结束 |
| `tags` | 子标签列表，每个标签包含 `name` 与 `max` |
| `remark` | 补充说明，当前 UI 文案为“账号详情 (数字id/小号/账号)” |
| `logs` | 每日打卡记录，key 为日期 |

---

## 2.3 logs 记录结构

当前代码支持两种记录形式：

### 1. 无标签任务

```json
{
  "2026-04-09": true
}
```

表示该任务在当天已完成。

### 2. 有标签任务

```json
{
  "2026-04-09": {
    "协战": 1,
    "勾协": 1,
    "寄养": 0
  }
}
```

表示当天每个标签各自完成了多少次，只有当所有标签都达到各自 `max` 时，该任务才视为“今日完成”。

# 三、任务状态系统

## 3.1 状态定义

```text
upcoming  → 未开始
ongoing   → 进行中
finished  → 已结束
```

---

## 3.2 判定逻辑

```js
if (start && today < start) return "upcoming";
if (end && today > end) return "finished";
return "ongoing";
```

说明：

* `start` 之前显示为 `upcoming`
* `end` 之后显示为 `finished`
* 若 `end` 为空，则任务会持续处于 `ongoing`

---

## 3.3 UI 展示规则

| 状态 | 展示位置 |
| --- | --- |
| `ongoing` | 展示在「今天（进行中）」 |
| `upcoming` | 展示在「即将开始」 |
| `finished` | 当前不展示 |

注意：`finished` 任务只是**不渲染**，并不是自动删除，数据仍然保留在本地存储中。

---

# 四、当前已实现功能

## 4.1 今日任务页

### 功能说明

* 展示所有 `ongoing` 任务
* 按分组顺序渲染任务列表
* 空分组不显示
* 每个任务支持查看名称、标签、备注、时间范围、完成状态
* 右下角固定悬浮按钮 `＋` 用于创建任务

---

## 4.2 即将开始列表

### 功能说明

* 展示所有 `upcoming` 任务
* 展示计划开始日期
* 可直接进入编辑
* 当前不支持在该区域直接打卡

---

## 4.3 新建 / 编辑任务弹窗

### 当前字段

* 游戏ID / 任务名称
* 任务类型标签
* 账号详情
* 分组
* 开始日期
* 结束日期（选填）

---

### 当前交互规则

* 新建任务时默认 `logs = {}`
* 编辑任务时会保留原有 `logs`
* 分组支持从下拉中选择，也支持直接新建
* 标签支持输入回车创建
* 标签支持从历史标签中快速补全
* 若重复添加同名标签，会增加该标签的 `max`
* 编辑态提供删除操作，并带确认弹窗

---

## 4.4 标签系统

当前实现中的 `tags` 不是普通文案标签，而是**带每日目标次数的执行子项**。

例如：

```json
[
  { "name": "协战", "max": 1 },
  { "name": "寄养", "max": 3 }
]
```

含义是该任务当天至少需要完成：

* `协战` 1 次
* `寄养` 3 次

只有两个标签都达标，整个任务才算完成。

---

## 4.5 打卡系统

### 无标签任务

行为：

```text
点击任务卡片 → 今日完成
再次点击任务卡片 → 取消今日完成
```

数据写法：

```js
logs[today] = true
```

---

### 有标签任务

行为：

```text
点击标签按钮 → 当前标签完成次数 +1
达到 max 后再次点击 → 该标签计数归零
```

注意：

* 有标签任务不能通过点击整张卡片直接完成
* 需要逐个标签操作
* 整体完成状态取决于所有标签是否都达到各自 `max`

---

## 4.6 分组系统

### 当前实现

* 分组用于组织「今天（进行中）」中的列表区块
* 分组顺序由 `groups` 数组决定
* 新建任务时如果选择“新建分组”，保存后会追加到分组列表
* 默认内置分组为：`健康`、`学习`、`游戏代肝`

---

# 五、关键业务规则

## 5.1 今日是否完成

当前逻辑分两类：

### 无标签任务

```js
logs[today] === true
```

### 有标签任务

```js
tags.every(tag => (log[tag.name] || 0) >= tag.max)
```

---

## 5.2 有标签任务的特殊行为

当任务带标签时：

* 卡片右侧的 `✅ / ⭕` 只是状态展示，不是点击入口
* 真正的点击入口是每个标签按钮
* 标签达到目标后会变成高亮状态
* `max > 1` 时，UI 会显示 `当前次数 / 目标次数`

---

## 5.3 编辑任务不会重置进度

当前保存逻辑只会更新这些字段：

* `name`
* `group`
* `start`
* `end`
* `tags`
* `remark`

不会清空原有 `logs`。

---

## 5.4 已结束任务的当前处理方式

已结束任务会从主界面消失，但：

* 不会自动删除
* 当前没有“历史任务”页面
* 当前没有恢复入口或归档视图

这意味着已结束任务仍在本地数据里，只是用户看不到。

---

# 六、与传统工具的区别

| 类型 | 核心特征 |
| --- | --- |
| ToDo | 通常是一次性完成 |
| Habit | 通常是无限周期、长期重复 |
| 本产品 | 在固定时间区间内进行每日执行追踪 |

当前版本进一步强化了一个差异点：

> 一个任务不仅可以“做 / 没做”，还可以拆成多个需要逐项完成的子标签。

这使它更适合描述游戏日常、账号代肝、多步骤日执行任务，而不是普通清单事项。

---

# 七、当前版本限制

以下能力在当前代码中**尚未实现**：

* 历史任务页 / 已结束任务页
* 完成率统计
* 连续打卡（streak）
* 提醒能力
* 周期任务（如每周执行）
* 多人协作
* 云端同步 / 后端接口

---

## 7.1 一个实现细节风险

当前“今天”的计算方式使用：

```js
new Date().toISOString().slice(0, 10)
```

这意味着日期基于 UTC 字符串截取，而不是显式按本地时区日历日计算。在跨时区或夜间场景下，可能出现“今天”与用户体感日期不完全一致的问题。

* 任务状态与日期判断集中在 `taskUtils`

# 九、总结

当前版本的 TimeTrackr / PhaseDo 本质上是：

> **一个面向阶段性日常执行的本地任务追踪器**

它解决的核心问题不是“我今天要做哪些一次性事项”，而是：

> **在一段时间内，我每天是否把这项任务及其子步骤执行到位了。**

与原始概念稿相比，当前实现已经明确具备以下产品特征：

* 单页结构，聚焦当天与即将开始
* 任务支持时间区间
* 支持分组管理
* 支持标签化子任务与次数目标
* 使用本地存储，轻量但暂无历史分析与云同步能力
